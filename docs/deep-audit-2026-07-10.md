# Deep Audit — StreamController `local-integration` (2026-07-10)

**Tree audited:** `local-integration` @ c54a6a24 (13 squashed topic commits ahead of `main`, merge-base 5eebf549).
**Method:** 12 parallel audit passes — 6 subsystem auditors (DeckController core, media layer, store/assets, pages/persistence, plugin system, app shell/GUI), 1 test-harness auditor, 5 commit reviewers covering all 13 branch commits — plus 2 contributed peer-session audits (AssetManager/Store reuse, per-scenario test deep-dive), spot-verification of single-source findings, and a full suite run.
**Baseline:** scenario suite **32/32 PASS** on the working tree. `[DBG-vmix]` instrumentation confirmed fully stripped; no debug leftovers introduced by the branch (the two stray `print()`s found are pre-existing on `main`).
**De-duplication:** findings already catalogued in `docs/memory-footprint-plan.md` (§6 bug appendix etc.) are *not* re-reported; where relevant they appear under "Known-issue status" with a verified fixed/open verdict.

**Origin flags used throughout:**
- `[branch]` — introduced by local-integration (commit sha given where it came from the commit review)
- `[pre-existing]` — present on `main` at the merge-base
- `[branch-amplified]` — pre-existing defect whose blast radius the branch increased
- `[peer]` — contributed by an external peer-session audit
- Confidence: *certain* (verified by reading/executing the exact path), *likely*, *speculative*. Items marked **REPRODUCED** were demonstrated at runtime by an auditor. Items marked **×N** were independently derived by N separate auditors.

## 1. Executive summary

The branch itself is in good shape: all 13 commit messages' claims verified against their diffs, no unbacked claims of substance, no debug leftovers, and the hard parts (single-writer pipeline, config-gen stamping, screensaver deferral, cache designs) are carefully reasoned and genuinely test-pinned. Most defects the commits shipped with were fixed by later commits in the same branch.

Four systemic themes dominate what remains:

1. **The sole writer thread is one exception away from a frozen deck.** `MediaPlayerThread.run()` has no exception guard (`@log.catch` commented out), and the branch promoted it from "one renderer among several" to the *only* thread that paints, sets brightness, clears, and closes the device. Three auditors independently converged on this; five concrete unguarded crash vectors were verified, two of them `[branch]` (B-01).
2. **Slot/lifecycle TOCTOUs survive at the edges of the new architecture.** The seq/generation machinery correctly serializes the queue level, but single-slot task fields, page-cache eviction, and `background_video` release are still read-then-act without revalidation (B-04, B-07, B-08, B-11).
3. **Persistence outside `Page.save()` is still crash-unsafe and quietly lossy.** The branch hardened exactly one writer; settings, page-manager metadata, migrations and importers still truncate-write, and three distinct pre-existing bugs *silently destroy user data* (B-06, B-09, B-10, B-14).
4. **Error visibility is the meta-bug.** No `sys.excepthook`/`threading.excepthook` anywhere, `@log.catch` swallowing structural failures (a *nonexistent* method call in the store-update path has never been noticed), and the new event dispatcher logs failures without tracebacks. Half the bugs in this report would have been field-visible years earlier with the one-time hook install (RD-01).

Counts: **9 HIGH / ~30 MED / ~50 LOW** findings; **~20 test gaps** (5 structural); **14 refactors**; **8 redesigns**. The single most valuable small change: a rate-limited try/except around the media-writer loop body. The single most valuable structural change: a pin-count ownership API for the page cache.

## 2. HIGH findings

### B-01 · Unguarded sole-writer thread: any render-path exception permanently freezes the deck — [pre-existing class, branch-amplified] · certain mechanism / likely occurrence · ×3
`src/backend/DeckManagement/DeckController.py:323-424` — `run()` has `# @log.catch` commented out; task bodies catch only `StreamDeck.TransportError` (:171, :204). After the single-writer migration this thread is the only path for paints, brightness, Clear and the terminal ClearAndClose; when it dies, `submit_control` keeps accepting (`_stop` never set) so `control_q` grows unbounded, and `close()` only escapes via timeout. Verified unguarded vectors:
- (a) `ScreenSaver.show()` swaps `inputs = {}` then rebuilds (ScreenSaver.py:163-165) while the tick loop direct-subscripts `inputs[Input.Key]` (:375, :381-382) — `KeyError` window every screensaver entry; the adjacent `_needs_key_ticks` deliberately uses `.get()`, these three don't. [pre-existing]
- (b) `Background.set_video(None)` (via `apply_prebuilt("blank")`) nulls `background.video` between the loop's None-check (:353) and five re-reads (:354-367). [pre-existing]
- (c) Touchscreen `get_current_image` derefs `active_page` with no guard (:4272-4276); `load_page(None)` and `close()` step 8 null it. [pre-existing]
- (d) **[branch, e314a086]** touchscreen bg-video tick gate None-checks `state.background_video` then re-reads `.fps` unlocked (:3849-3854) while `_release_background_video()` (:4261-4265) nulls it under `_background_video_lock` from compositing threads — an ordinary "swap video → image in sidebar" can kill the writer. *Spot-verified.*
- (e) **[branch, 0ad97515]** `_exec_clear` reads `touchscreen_task.submit_seq` via two unlocked reads (:524-526) while `clear_media_player_tasks` (:1710) / `close()` step 7 (:1820) null the slot — trigger is exactly the screensaver-show + load_page combination.
**Fix shape:** rate-limited try/except around the loop body (log + continue), single-snapshot reads for (d)/(e), build-then-swap for the screensaver input dict. The `__init__` comment at :836-841 already acknowledges the loop is unguarded.

### B-02 · mp4 tile-cache builder thread spins at 100 % CPU forever when a build cannot complete — [branch, b05cce20 P2.1] · certain · **REPRODUCED ×2**
`src/backend/DeckManagement/Subclasses/mp4_tile_cache.py:538-551` — `_run_builder` loops `get_frame(last+1)` with no sleep; when `_end_of_source()` releases `self.cap` without `_complete` (VideoWriter open failure :239-242, `os.replace` OSError :371-373, reopened cache not opened :375-376, or truncated mp4 with positive frame metadata), every `get_frame` returns instantly and the loop becomes a pure spin until refcount-0 — i.e., for as long as the video key is on screen. Reproduced: **4,677,116 calls/sec** with a stubbed dead writer. Fix: treat (`cap is None and not _complete`) as terminal in the builder loop. Zero test coverage for this whole failure family (TG-04).

### B-03 · `InputImage` re-decodes from disk on every composite for sub-tile sources, and closes the image under concurrent readers — [branch, b05cce20 P2.4] · certain · **REPRODUCED ×2**
`src/backend/DeckManagement/Subclasses/KeyImage.py:102-147` — the "needs more resolution" check (:124) can never be satisfied when the source itself is smaller than the composed size (any 64 px icon on 72 px tiles; any wide/tall image whose fitted minor dimension < tile). Every `get_raw_image()` re-runs `Image.open` + `convert("RGBA")` + enhance, then `old_image.close()` closes the previously returned image out from under the media thread ("Operation on closed image" reproduced). On a background-video page this is per-frame disk I/O + decode + a hot use-after-close race; the fg-cache does not shield it (it skips only the resize). Fix: memoize the "source cannot satisfy" outcome (clamp `needed` to native size) and stop closing the swapped-out image (refcounting is the stated policy at DeckController.py:2082-2084).

### B-04 · Page-cache eviction can gut a live page (use-after-evict family) — [branch, 977b79b6 + f0bf28a8 interplay] · likely · ×2
`src/backend/PageManagement/PageManagerBackend.py:157-195` snapshots evictable pages under `_pages_lock`, then runs `clear_action_objects()` + `pop` outside it (deliberately, so plugin hooks can't wedge the lock). Three windows where a page passes the guards but is live:
1. **Screensaver-pending page** — `_screensaver_pending_page` (DeckController.py:1412) is non-active, `ready_to_clear=True`, held for potentially hours, and invisible to the eviction guards. Evicted → `hide()` loads a page whose every action is dead, and the next `get_page` mints a duplicate `Page`.
2. **Activation TOCTOU** — between snapshot and out-of-lock gutting, `get_page`+`load_page` can activate the page; eviction then tears down the *active* page's actions (`initialize_actions` skips them — `on_ready_called` already True). Most plausible under WindowGrabber cycling > `max_pages`, which is exactly the workload that constantly evicts.
3. Same window for fetched-not-yet-loaded pages (`move_page`'s per-controller instantiations generate the triggering cache pressure themselves).
**Fix:** re-validate under `_pages_lock` immediately before each teardown (skip if now active or pending), pop *before* `clear_action_objects` so `get_page` can't re-serve a condemned page — or the pin-count API (RD-02), which closes all three structurally.

### B-05 · Global single-lane event dispatcher: one wedged plugin observer stalls all plugin events app-wide, queue unbounded — [branch, b05cce20 P1.4] · likely · ×2, *spot-verified*
`src/backend/PluginManager/event_dispatch.py:56, 87-96` — `_dispatch_executor = ThreadPoolExecutor(max_workers=1)`; every `EventHolder.trigger_event()` and AssetManager `Observer.notify()` for every plugin funnels through one worker with head-of-line blocking. On `main`, observers ran in the caller's thread — a blocked observer hurt only its own source. Now one wedged observer (this codebase has a real pulsectl-wedge precedent) permanently stalls event delivery for every plugin, and the executor's `SimpleQueue` grows for the process lifetime (AudioControl fires tens of events/sec) with no stuck-watchdog (the tick path has a >10 s one at DeckController.py:3084; this path has none). The single-thread design is deliberate (persistent asyncio loop) — the missing pieces are a watchdog, bounded/per-holder lanes, and documentation of the cross-plugin coupling.

### B-06 · Icon/wallpaper "update" deletes the installed pack before downloading; failure leaves it gone — [pre-existing pattern; trigger enabled by branch c54a6a24 + ed4fd058] · certain mechanism / likely trigger · ×2
`src/backend/Store/StoreBackend.py:1035-1069` — `install_icon`/`install_wallpaper`/`install_sd_plus_bar_wallpaper` each `rmtree` the installed pack *before* `download_repo`, and ignore its return. `install_plugin` got exactly this hardening on the branch (:928-933); these three did not. The branch made the bad path *reachable offline*: the new stale-cache fallback lets startup auto-update (`main.py:232`, default on) load catalogs from cache with no network, walk into the uninstall-then-download flow, and — since ed4fd058's falsy return no longer aborts the loop — consume every pending item. Result: packs deleted, keys referencing them break, and `local_sha=None` means it isn't retried. Realistic trigger: 429-throttled update (observed live on this IP) or offline launch with a warm cache.

### B-07 · Exception in store install/uninstall wedges all store downloads for the session — [pre-existing] · certain · ×2
`src/windows/Store/Preview.py:189-207` — `@log.catch` eats any exception after `currently_downloading=True` is set; the reset never runs, and every later download click spins forever in `while ...: time.sleep(0.1)`. Confirmed live feeder: `get_last_commit` (`StoreBackend.py:246-256`) is outside the `NoConnectionError` contract, so installing a branch-pinned plugin offline raises straight into this wedge. The flag check/set is also non-atomic (double-click → concurrent installs — the corruption the flag exists to prevent).

### B-08 · Touchscreen single-slot task races lose frames on static content — [branch, 9a54e1b6/0ad97515] · certain mechanism / timing-dependent
Two shapes: (a) `perform_media_player_tasks` reads the slot then unconditionally nulls it (DeckController.py:677-679) — a producer assigning in between loses its frame *and* `_last_enqueued_hash` is left set without `_last_img_hash`, so a static strip stays stale indefinitely; (b) `_exec_clear`'s per-key `get`-then-`del` (:520-523) can delete a newer task whose `submit_seq` contractually survives the clear — key stays blank on static content. Animated content self-heals next tick; static content has no next tick. Fix: identity-compare before null/del.

### B-09 · Page import & duplicate are completely dead (`UnboundLocalError` on every use) — [pre-existing] · certain · *spot-verified*
`src/windows/PageManager/elements/MenuButton.py:148-159` — `page_name = os.path.splitext(os.path.basename(page_path))[0]` reads `page_path` before its assignment two lines later; the user's chosen `name` parameter is ignored. Both "Import → Page" and "Duplicate page" route here and fail unconditionally; the exception fires inside a Gio action callback, so it lands terminal-only (see RD-01) and the UI silently does nothing. Fix: `page_name = name`. Zero test coverage.

### B-10 · Onboarding builds GTK widgets on a worker thread — [pre-existing] · certain path / likely crash
`src/windows/Onboarding/PluginRecommendations.py:44-71` — `load()` constructs `PluginRow` widgets (Adw.ActionRow + CheckButton) and calls `group.add(row)` on a plain thread; `OnboardingWindow.py:324-331` reads `check.get_active()` from another worker. Fires on every first launch while the store fetch races the user swiping the carousel — the exact documented off-main-GTK crash class. Related, same class: store pages construct `PluginPreview` widget trees on loader threads (`PluginPage.py:67` pattern ×4), `set_visible_child` off-main (`StorePage.py:135`), and `CustomAssetChooserFlowBox` + 50 `AssetPreview`s built on a worker (`Chooser.py:47`).

## 3. MED findings (by subsystem)

### DeckController / rendering
- **B-11** [branch] · likely — **Opaque-color static keys never get their initial device paint after switching onto a background-video page.** `update_all_inputs` skips key device writes when a bg video is present (:953-972, pre-existing) *and* the branch's per-frame optimization skips alpha==255 keys (:3455-3459); main's wasteful unconditional repaint was incidentally the initial paint. Device shows the *previous page's* content on those keys until a keypress or plugin `on_update` — the inverse of the preview-divergence bug b57cb0ba fixed. Masked in the field because most opaque keys carry actions. Fix: in the bg-video branch still `update()` alpha-255 keys (their tile is hidden — they cannot disturb the video). The render-design doc's "self-heals within one tick" claim is false for this case.
- **B-12** [branch] · certain path — **`close()` step 6 runs plugin teardown hooks unbounded** (:1804-1805 → per-page `clear_action_objects` with no deadline). A wedged hook (pulsectl precedent) strands the close *before* steps 7–9: media sweep, fallback `deck.close()`, and `discard_controller` all skipped — the unplug leak reintroduced — and `_closing=True` makes retry a no-op forever. Fix: bounded join, or deregister (step 8) before hooks (step 6).
- **B-13** [branch vs. main tip] · certain — **Unconditional `gc.collect()` per page switch** (:1500) plus two more in `Background.set_image`/`set_video` (:1940, :1958) — 2-3 full collections per switch, often on the GTK thread; tens of ms each on a multi-hundred-MB heap. Upstream 88d632cc (one commit past merge-base) added `maybe_collect_garbage()` with a 10 s floor *for exactly this* and **will conflict at merge** — adopt the rate-limited form deliberately, not by conflict resolution. Related merge-reconciliation item: the branch's `set_brightness` always writes where upstream now skips identical values (deliberate? a dedup would break resume recovery since `_run_pending_repaint` doesn't re-send brightness — decide explicitly).
- **B-14** [branch, e314a086] · certain — **`Page.set_media_fps` live-apply has no active-page guard** (`Page.py:990-999`): iterates matching inputs across *all* controllers (unlike `update_input` :746), so editing one page's FPS row rebases the playing video timeline on another deck showing a different page. - **B-15** [branch, b05cce20] · likely — **close-vs-load background-video leak**: `close()` neither bumps `_page_load_generation` nor cancels `_bg_future`, so a racing `load_page` can attach a fresh `BackgroundVideo` (capture + registry ref + builder thread) *after* the close's resource sweep — leaked until process exit.
